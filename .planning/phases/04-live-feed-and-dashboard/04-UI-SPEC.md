Source: measured from RecordingsPage.axaml, RecordingDetailPage, SettingsPage.axaml — all prior phases use a consistent 4-size scale.

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Body | 14px | Regular (400) | 1.5 |
| Label / caption | 12px | Regular (400) | 1.4 |
| Heading | 18px | SemiBold (600) | 1.2 |
| Display (section title) | 24px | SemiBold (600) | 1.2 |

Notes:
- 14px body is the dominant size used in all card metadata fields across prior phases
- 12px label is used for metadata captions, sub-labels, and time codes
- 18px heading is used for section titles (e.g. "Clips in this trip", "Could not load settings")
- 24px display is used for page-level titles and prominent empty-state headings

## Interaction Contracts

### Live Feed Page

1. **On page appear (tab navigation to Live Feed):** immediately call `StartLiveFeedCommand` — do not wait for user input. Show loading placeholder during stream startup.
2. **On page disappear (tab navigation away):** call `StopLiveFeedCommand` — release media player resources. VideoView must not consume GPU resources off-screen.
3. **Camera toggle:** tapping the inactive segment calls `ToggleCameraCommand` with `CameraId` parameter. Active segment updates immediately (optimistic UI — flip `SelectedCamera` before stream confirms). Stream restarts automatically using the new camera URI.
4. **Connection loss:** `IsConnectionLost` observable property on LiveFeedViewModel drives `IsVisible` swap between VideoView and connection-loss placeholder. Segmented toggle remains enabled so user can attempt the other camera.
5. **Retry button:** calls `RetryCommand` — attempts to restart the stream for the currently selected camera.

### Dashboard Page

1. **On page appear:** load dashboard data if not already loaded (`IsDashboardLoaded` guard prevents reload on every tab visit).
2. **Device disconnect while on Dashboard:** items remain visible (stale data is acceptable) — do not clear and re-show empty state on disconnect.
3. **Tap compact card (Recording or Trip):** navigate to RecordingDetailPage using the same `INavigationService` pattern as RecordingsViewModel — push page onto the Recordings-tab navigation stack and switch to Recordings tab (tab index 1).
4. **"See All" — Recent Recordings:** switch to Recordings tab (tab index 1), clear any active filter.
5. **"See All" — Recent Trips:** switch to Recordings tab (tab index 1), clear any active filter (trips appear inline in the unified recording list).
6. **"See All" — Recent Events:** switch to Recordings tab (tab index 1), apply `EventType.GShock` filter (events section shows highest-severity events — G-Shock is the canonical "event" type). Source: D-12 in CONTEXT.md.

### Cross-Tab Communication

`AppShellViewModel` is the mediator for "See All" navigation. `DashboardViewModel` must receive either an `Action<int>` callback or a direct reference to `AppShellViewModel` to trigger tab switches and filter application. The same constructor-injection pattern as Phase 1 `ManualConnectionViewModel` applies here (CONTEXT.md accumulated decisions: "ManualConnectionViewModel constructed manually by AppShellViewModel with onClose Action callback").

## Responsive Breakpoints

Source: Phase 1 decision — `AdaptiveBehavior` breakpoint at 600px width.

| Width | Class | Layout impact |
|-------|-------|---------------|
| < 600px | `portrait` | Video area 300px fixed height; camera toggle below video; dashboard single-column card list |
| >= 600px | `landscape` | Video area fills remaining height (`*`); camera toggle pinned below video; dashboard may use 2-column card grid if space permits (Claude's discretion — single-column is acceptable for v1) |

For v1: single-column layout is used on both breakpoints for dashboard cards. The responsive grid expansion is deferred — it requires `AdaptiveClassSetter` style classes which add XAML complexity without proportional user value at this stage.
